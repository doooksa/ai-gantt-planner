# AI assistants usage log

Ведётся по ходу разработки. Из этого файла собирается раздел README
«AI assistants usage»: что делегировано ассистенту, что модель предложила
неверно и как исправлено, что сделано руками.

## Эксперимент — конфигурируемый LLM-слой + оценка free-tier провайдеров

- Сделан **конфигурируемый OpenAI-совместимый профиль**: `LLM_BASE_URL` +
  `LLM_API_KEY_ENV` + `LLM_MODEL` (дефолт — OpenRouter/Haiku, переключение на
  Google — env-только). Ключ провайдера читается по имени переменной из
  `LLM_API_KEY_ENV` (косвенность: `OPENROUTER_API_KEY` vs `GOOGLE_API_KEY`).
- Разобрана авторизация OpenRouter (мёртвый ключ → «User not found»; низкий лимит
  ключа → `402 "can only afford N tokens"`, из-за того что агент не ограничивал
  `max_tokens`; починено капом 2048).
- **Оценка free-tier (честно, не приняты):** gemini-2.5-flash быстр (~2–8 с/команда)
  и корректно зовёт инструменты, но free tier = **~5 req/min + ~20 req/day** на
  модель. Команда агента = несколько запросов → дневная квота даёт ~5 правок в
  сутки; гейт 5×10 (~200 запросов) не проходит структурно. OpenRouter free-модели
  — rate limit 429. Вывод: остаёмся на платной Haiku; для Google в прод — платный
  тариф (переключение тем же env). Добавлена фикстура `GATE_PAUSE_SECONDS` для
  пейсинга гейта под минутную квоту (дневную она не обходит).

## Phase 1 — домен без UI (models, scheduler, validators, patches, Excel, storage)

**Assistant:** Claude Code (Opus 4.8).

### Что делегировано ассистенту
- Полная генерация доменного слоя backend'а по спецификации из `CLAUDE.md`:
  - `domain/models.py` — Pydantic v2 модели (Task/Plan/ScheduledTask, Patch/Op/Selector, Diff).
  - `domain/scheduler.py` — детерминированный планировщик: DFS-топосорт с
    извлечением пути цикла, forward pass по календарным дням.
  - `domain/validators.py` — структурная валидация + русские сообщения об ошибках.
  - `domain/patches.py` — атомарное применение патчей, селекторы
    (by_id/by_name/by_assignee), diff before/after.
  - `domain/slug.py` — генерация id из имён с транслитерацией кириллицы.
  - `excel.py` — импорт/экспорт openpyxl, нормализация заголовков, резолв
    предшественников по имени, ошибки с номером строки.
  - `storage/` — SQLite singleton-план + стек снапшотов для undo, seed.
- Полный набор pytest-тестов (39 шт.), покрывающий обязательный список гейта:
  цепочка, ромб, цикл→ошибка, несуществующий предшественник, кириллические
  заголовки с лишними пробелами, пустые предшественники, невалидная длительность.

### Ключевые проектные решения, принятые в диалоге
- **`offset_days` на `Task`.** Команда «перенеси задачу на N дней позже» конфликтует
  с принципом «даты — derived, не хранятся». Решение: добавлено опциональное поле
  `offset_days` — это *вход* планировщика (как `duration_days`), а не хранимая дата.
  Планировщик по-прежнему вычисляет start/end; инвариант соблюдён. Задокументировано
  в коде и предназначено для роадмапа.
- **Диапазон дат — включительный:** `end = start + duration_days - 1`
  (задача длительностью 1 начинается и заканчивается в один день).
- **Удаление задачи** не запрещается при наличии зависимых: ссылки на удалённую
  задачу вычищаются, а в `diff.warnings` добавляется предупреждение (агент затем
  формулирует его пользователю).
- **Атомарность патча** реализована через применение к глубокой копии + финальная
  валидация/планирование; при любой ошибке исходный план не трогается.

### Что модель предлагала неверно / что пришлось поправить руками
- Первый вариант теста `test_shift_task` вызывал планировщик через
  `__import__("app.domain.scheduler", ...)` — некрасиво; заменено на обычный
  `from app.domain.scheduler import schedule`.
- Коммит-сообщения: ассистент по ошибке применил PowerShell-синтаксис here-string
  (`@'...'@`) внутри Bash-инструмента — в первый коммит попали лишние строки `@`.
  Исправлено `git commit --amend -F -` с корректным heredoc.

### Сделано руками / проверено человеком
- Прогон `pytest` (гейт Фазы 1): **39 passed**.
- Сгенерирован и проверён на реимпорт `examples/sample_plan.xlsx`.
- Git-история: init, conventional commits по логическим шагам, локальная
  идентичность репозитория, `.gitignore` без секретов.

## Phase 1 — доработки по ревью (offset_days: round-trip и сдвиг раньше)

По итогам ревью Phase 1 закрыты две «протечки» вокруг `offset_days`.

### 1. Excel round-trip терял `offset_days` молча
**Решение (осознанное ограничение, а не «тихий баг»):** формат обмена жёстко
зафиксирован спекой — 5 структурных колонок + 2 *вычисленные* даты. Ручной сдвиг
(`offset_days`) не является колонкой, поэтому export→import сбрасывает его в 0.
Экспортированные даты сдвиг учитывают, но при импорте даты игнорируются (они
derived, не source of truth) — значит реимпортированный план считается без сдвига.
- Задокументировано в docstring `excel.py` (раздел «Known limitation»).
- Зафиксировано тестом `test_export_import_drops_offset_days` как контракт.
- Альтернатива (скрытая колонка «сдвиг, дн.» для полной точности) вынесена в
  roadmap. Вариант «восстанавливать offset из дат при импорте» отвергнут: хрупко
  и нарушает принцип «даты derived».

### 2. «Перенеси на N дней раньше» молча клампился в 0
Раньше `shift_task` делал `max(0, offset+days)` — часть сдвига тихо терялась.
**Решение:** отрицательный `offset_days` остаётся запрещённым инвариантом
(`ge=0`) — старт раньше самой ранней допустимой даты сломал бы зависимости.
`shift_task` раньше допустим только чтобы отмотать предыдущий сдвиг вперёд; выход
за 0 → явный отказ `PlanValidationError(code="shift_before_earliest")` с понятным
русским сообщением и указанием максимально доступного сдвига. Атомарность патча
сохраняется. Тесты: `test_shift_earlier_unwinds_previous_forward_shift`,
`test_shift_earlier_below_zero_is_refused`,
`test_shift_earlier_partially_out_of_range_is_refused`.

**Гейт после доработок:** `pytest` — **43 passed**.

## Phase 2 — MCP + агент (сервер, bridge, агентский цикл, REST/SSE/WS)

**Assistant:** Claude Code (Opus 4.8).

### Что делегировано ассистенту
- `mcp_server/` — FastMCP-сервер с ровно 4 инструментами (`get_plan`,
  `validate_patch`, `apply_patch`, `undo_last`), смонтирован в тот же
  FastAPI-процесс по streamable HTTP на `/mcp`.
- `llm/mcp_openai_bridge.py` — конвертация MCP-схем → OpenAI tools format и
  сериализация результатов инструментов обратно в сообщения.
- `llm/agent.py` — агентский цикл (≤10 итераций, get_plan → validate → apply),
  транспортонезависимый: принимает готовую `ClientSession` и OpenAI-совместимый
  клиент.
- `services.py` — единый слой мутаций для REST и MCP (чтобы поведение «сделал
  агент» и «нажал пользователь» не расходилось), с бродкастом `{version, diff}`.
- `api/` + `main.py` — REST (`/api/plan|upload-excel|export-excel|undo|
  reset-demo|health`), SSE `/api/chat`, WebSocket `/ws`, CORS из `FRONTEND_ORIGIN`.
- `events.py` — in-process шина для WebSocket-бродкаста.
- Оффлайн-тесты (20 шт.): MCP-инструменты через in-memory transport, bridge,
  агентский цикл с фейковым LLM, REST/SSE через TestClient. Плюс живой гейт
  `tests/test_agent_scenarios.py` (10 команд, skip без ключа).

### Ключевые проектные решения
- **Транспортонезависимый агент.** `run_agent(session, llm, ...)` не знает про
  HTTP — это позволило покрыть весь цикл оффлайн (in-memory MCP + фейковый LLM),
  а в проде `/api/chat` соединяется с собственным `/mcp` по streamable HTTP.
- **Единый `services.py`** вместо дублирования логики в MCP-инструментах и REST.
- **Ошибки инструментов — структурой `{ok:false, error, code}`**, а не
  исключением: агент читает их и сам исправляется (покрыто тестом
  `test_agent_recovers_from_invalid_patch`).
- **SSE — событийный** (`tool`/`applied`/`message`/`done`), не по токенам.
  Потоковый вывод финального текста по токенам вынесен в roadmap.

### Что модель предлагала неверно / что пришлось поправить руками
- **Монтирование MCP:** `streamable_http_app()` сам обслуживает путь `/mcp`;
  наивный `app.mount("/mcp", ...)` дал бы `/mcp/mcp`. Исправлено:
  `mcp.settings.streamable_http_path = "/"` перед монтированием в `/mcp`.
- **`session_manager.run()` — один раз на экземпляр.** В тестах `with TestClient`
  повторно входил в lifespan → RuntimeError. Решение: API-тесты не используют
  lifespan (chat подменяет провайдер MCP-сессии на in-memory).
- **SQLite и потоки.** TestClient обслуживает запросы из worker-потока →
  `ProgrammingError`. Добавлен `check_same_thread=False` (доступ фактически
  сериализован одним event loop).

### Сделано руками / проверено человеком
- Оффлайн-гейт: **63 passed, 10 skipped** (live).
- Живой рантайм-смоук (uvicorn): health, план с датами от сегодня, экспорт xlsx,
  ответ `/mcp`, бродкаст по WebSocket `{version, diff}`, и полный round-trip
  реального MCP-клиента к `/mcp` по streamable HTTP (4 инструмента, get_plan → 8
  задач).
- **Не проверено:** живой гейт из 10 команд через реальный LLM — требует
  `OPENROUTER_API_KEY` (в окружении нет; прогон запланирован с ключом
  пользователя).

### Что делегировано ассистенту

### Окружение (сказал, не обошёл — затем закрыл по согласованию)
- Node.js в окружении не было. По согласованию с пользователем поставил **portable
  Node 20.20.2** в папку вне репозитория (`task/.tools/`), не в систему; собрал и
  запустил фронт, прогнал демо-сценарий в браузере превью.

### Что модель предложила неверно / что пришлось поправить руками
- **SVAR Gantt рендерил только 2 задачи из 8.** Диагностика через DOM/фибер
  показала: данные (8 задач + 9 связей) доходят до компонента, но виртуальный
  скроллер SVAR измерял высоту вьюпорта как 0 при монтировании (ResizeObserver в
  встроенном браузере превью). Починка: форс-высоты для тем SVAR
  (`.wx-theme/.wx-gantt` = 100%) и внутренних контейнеров вьюпорта — после этого
  таймлайн отрисовал все 8 баров.
- **Левая таблица-грид SVAR** (`@svar-ui/react-grid`) продолжала недомерять свой
  вьюпорт и отдавала ~2 строки с несовпадающей высотой. Решение: рендерить Gantt
  как таймлайн с подписями на барах (`columns={false}`), детали — в `TaskModal`
  по клику на бар. Возврат таблицы — правка в одну строку.
- `select-task` не срабатывал на синтетический `.click()` (SVAR слушает
  pointer-события) — проверял реальным кликом через инструмент превью.

### Сделано руками / проверено человеком (демо-гейт)
Полный сценарий прогнан в запущенном приложении (uvicorn :8000 + Vite :5173):
- загрузка → все 8 задач на диаграмме, WS «online», версия v0;
- чат «Увеличь длительность Design до 5 дней» → стрим ответа + блок Applied
  changes (каскад сдвигов) → диаграмма и версия обновились через WS → Undo вернул;
- чат «Все задачи Ivan переназначь на Maria» (ввод в UI) → Applied changes
  (Backend API, AI Agent: Ivan→Maria) → Undo из блока вернул v0;
- клик по бару «Backend API» → TaskModal: Ivan, 4 дн., 2026-07-11…14,
  предшественник Design, зависимые AI Agent + Excel Export.

**Гейт Phase 3 (локально работает полный демо-сценарий): пройден.**

### Правки после проверки пользователя в реальном Chrome
Пользователь подтвердил: в настоящем Chrome виртуализация SVAR работает, все 8
задач рендерятся — баг был именно в среде превью. По итогам:
- убраны preview-only CSS-хаки (форс-высоты внутренних контейнеров SVAR);
- **возвращены колонки** таблицы (задача / исполнитель / длительность);
- **формат шкал** переведён со строк на функции date-fns + ru-локаль (строковые
  токены SVAR — не date-fns, поэтому раньше показывалось литеральное «MMMM
  yyyy»/«d»); неделя — дата начала недели, день — число;
- задан явный `start`/`end` (±2 дня вокруг плана) — таймлайн заполняет вид без
  пустых полей (шкалы день/неделя);
- промпт агента: краткий итог в 1–2 фразы, без дублирования списка изменений
  (детали — в блоке «Applied changes»).
Проверено: колонки и данные рендерятся; краткость ответа агента подтверждена
(итог ~165 символов, без по-задачного списка; в Applied changes 7 строк diff).
Финальную визуальную проверку шкал делает пользователь в Chrome.

## Фаза деплоя — конфигурация доставки (Vercel + Render)

### Делегировано Claude Code
- Dockerfile бэкенда (python:3.12-slim, uvicorn с биндом на `$PORT` для Render) и
  фронта (multi-stage node-build → nginx), `.dockerignore` для обоих.
- `nginx.conf`: SPA-fallback + reverse-proxy `/api`, `/ws`, `/mcp` на бэкенд —
  браузер работает same-origin, как в dev-прокси Vite.
- `docker-compose.yml`: обе службы, volume для SQLite, healthcheck; в env бэка
  `DATABASE_URL=sqlite:////data/gantt.db` (в volume).
- `render.yaml` (Blueprint, Docker web service, `healthCheckPath: /api/health`,
  `OPENROUTER_API_KEY`/`FRONTEND_ORIGIN` как `sync:false`) и
  `apps/frontend/vercel.json` (framework vite + SPA rewrites).
- Прод-CORS: `FRONTEND_ORIGIN` теперь парсится в список через запятую
  (`deps.py` → `frontend_origins`, `main.py` отдаёт его в `allow_origins`) — под
  прод-URL + preview-URL Vercel.
- Обработка холодного старта Render на фронте: в сторе — `bootstrap()` с ретраями
  и бэкоффом и флагом `waking`; в `App.tsx` — экран-лоадер «Сервер просыпается…»
  с подсказкой про ~30 сек; авто-reconnect WebSocket (2 с) уже был.
- Документация: `docs/ROADMAP_TO_PRODUCTION.md`, `docs/DEPLOY.md` (пошаговый
  клик-гайд Render→Vercel→CORS→smoke-test), обновлены README (§5 compose, §6
  Deployment) и STATUS.

### Что модель предложила/учла и как поправлено

  добавлен приём обоих (алиас) + срез хвостового слэша, чтобы `${BASE}/api/...`
  не задваивался.

  «WS через wss в проде» не понадобилась, зафиксировано в доках.

### Сделано/проверено руками (в этой среде)
- **Пересоздан `.venv`**: старый указывал на несуществующий после переезда
  Python 3.12; создан заново на 3.13.2, переустановлены зависимости.
